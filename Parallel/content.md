# 常见的一些并行策略
> 参考文章：https://blog.csdn.net/xuebinding/article/details/150708560

首先我们要知道目前主流的推理框架、推理方法中有哪些并行策略（ai 统计的）：

| 方法         | 针对痛点                                      | 适用场景 (硬件拓扑)                                 |
| :--------- | :---------------------------------------- | :------------------------------------------ |
| 数据并行 (DP)  | 模型能装进单卡，但数据量大，需加速训练或提升推理并发。               | 单机多卡 或 多机多卡  <br>(每张卡/每组卡放一个完整模型副本)         |
| 张量并行 (TP)  | 单层矩阵太大，单卡显存存不下权重或中间激活值。                   | 单机多卡  <br>(通常限制在同一台机器内的8卡/16卡之间，依赖高速NVLink) |
| 流水线并行 (PP) | 模型层数太深，整机显存都装不下，需按层切分。                    | 多机多卡  <br>(将不同层的计算分布到不同的机器上)                |
| 序列并行 (SP)  | 长文本导致激活值显存爆炸，且需配合TP降低通信开销。                | 单机多卡  <br>(通常与TP绑定，在同一台机器内切分序列维度)           |
| 专家并行 (EP)  | MoE模型中专家总数太多，单卡/单机存不下所有专家参数。              | 多机多卡  <br>(将不同的专家组分布到不同的机器或卡上)              |
| 上下文并行 (CP) | 超长上下文（如100万token）导致KV Cache和注意力计算无法在单机完成。 | 多机多卡  <br>(跨机器切分序列长度，通常用于极端长文本场景)           |
| 3D 并行      | 单一策略无法满足千亿/万亿参数模型的训练需求。                   | 大规模集群 (多机多卡)  <br>(典型组合：机内做TP/SP，机间做PP/DP)  |
