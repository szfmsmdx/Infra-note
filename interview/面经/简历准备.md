# nano-vllm-extend
### 项目整体概述
nano-vllm 运行流程：
- LLM 引擎初始化
	- 配置文件与多线程启动
		- 加载 config
		- 创建 shm
	- model runner初始化
		- 创建 PD 通信组
		- 加载模型权重
		- kv cache显存分配
		- warm up捕获cuda graph
- generate 启动引擎推理
	- seq prompt 通过 tokenizer 转化为 token_ids 包装为 Sequence 对象
	- 加入 scheduler 的 waiting 队列
	- 调度 prefill_seq 和 decode_seq
		- decode 优先，显存不足先抢占已经被加入 decode_seq 队列的
		- 再 prefill 调度，如果是新请求，给 seq 初始化 page attention 物理块
	- PD 分离执行推理 (engine.step)
		- 异步 decode 推理
		- 同步 prefill 推理
		- 收集 decode 结果，此时同步起到了 barrier 的效果，确保 decode 能够推完（拿到本次 decode 的结果）
		- 当你 prefill 推完最后一个 chunk prefill 块的时候才推送 kvcache
	- postprocess
		- 更新 sequence 的 num_cache_token
		- 检查是否 chunk 推完转换状态为 running

## engine.step 具体过程  
### 第一阶段：请求调度 `self.scheduler.schedule()`

`step()` 的第一步是向调度器询问当前需要执行哪些序列。进入子函数 `Scheduler.schedule()`，其逻辑分为解码调度和预填充调度两部分：
1. **Decode（解码）调度优先**：
    - 调度器首先遍历 `self.running` 队列（正在生成 Token 的请求）。
    - 对于每个请求，调用 `self.block_manager.can_append(seq)` 检查 KV Cache 的物理显存池中是否还有足够的空间追加这一个 Token。
    - **注意点（Preemption/抢占机制）**：如果显存不足（OOM），调度器会触发抢占机制 `self.preempt(seq)`。它会优先将刚刚加入 Decode 队列的其他序列踢回 `waiting` 队列，并调用 `block_manager.deallocate(seq)` 释放其 KV Cache 物理块。
    - 如果空间足够，调用 `block_manager.may_append(seq)`（在跨越 Block 边界时分配新物理块），并将序列加入本轮的 Decode 名单中，直至达到 `max_num_seqs` 上限。
2. **Prefill（预填充）调度与 Chunking（分块）**：
    - 接下来遍历 `self.waiting` 队列。为了兼顾长短请求，代码使用 `sorted(self.waiting, key=lambda s: len(s) - s.num_cached_tokens)` 按照剩余需要计算的长度对等待队列进行排序（短任务优先）。
    - **注意点（Chunked Prefill）**：这是大模型推理优化的关键点。它并没有一次性计算整个长 Prompt，而是通过 `chunk_len = min(remain_len, self.chunk_size, total_budget_remain)` 计算本次能够处理的 Token 数。其中 `total_budget_remain` 受到 `max_num_batched_tokens` 的严格限制。
    - 如果是首次调度，会调用 `block_manager.allocate(seq)` 为其计算 Prefix Cache Hash 并分配初始物理块。
    - 只有当本轮能够彻底处理完该序列剩余的所有 Token (`seq.num_cached_tokens + chunk_len == len(seq)`) 时，它才会被移出 `waiting` 队列。
### 第二阶段：模型执行 `ModelRunner.run()` 与 PD 分离架构
拿到 `prefill_seqs` 和 `decode_seqs` 后，引擎开始派发计算任务。这里的逻辑根据是否开启了 `pd_separation`（PD 分离）而产生巨大分歧。
**分支 A：标准模式（未开启 PD 分离）**
- 引擎无法同时进行 Prefill 和 Decode。它通过 `is_prefill = len(prefill_seqs) > 0` 决定当前是 Prefill 阶段还是 Decode 阶段（Prefill 具有绝对优先级）。
- 直接通过跨进程调用 `self.model_runner.call("run", seqs, is_prefill)` 启动推理。

**分支 B：PD 分离模式（核心亮点）**
- **异步发射 Decode**：主进程通过 `_drive_decode_runner("run", decode_seqs, wait_ack=False)` 利用 SharedMemory 将 Decode 任务指令下发给 Decode Worker 进程池。注意这里 `wait_ack=False`，意味着主进程下发命令后**不等待** Decode 进程确认，直接往下走以实现最大化并行。
- **同步执行 Prefill**：主进程接着调用 `self.prefill_runner.call("run", prefill_seqs, True)`，让 Prefill Worker 同步开始计算。
- **结果收集与屏障**：由于 Decode 计算通常比 Prefill 快，主进程通过 `self.prefill_runner.recv_decode_res` 阻塞等待并接收 Decode Worker 发回的生成 Token。这里巧妙地充当了双阶段的同步屏障。
- **注意点（P2P KV Cache 传输）**：如果某个 Prefill 请求在本轮完成了所有 Chunk 的计算（意味着它的 Prefill 彻底结束，下一轮将进入 Decode），引擎会收集其 `block_table`。然后向两边进程发送 `transfer_kv_cache` 指令。
    - 进入 `ModelRunner.transfer_kv_cache` 子函数：这里使用了一个专属的 CUDA Stream (`self.transfer_stream`) 和 NCCL 异步通信（`dist.isend` / `dist.irecv`）。这样 KV Cache 从 Prefill 节点拷贝到 Decode 节点的过程中，计算流完全不会被 CPU 阻塞，实现了计算与通信的完美掩盖 (Overlap)。

**子函数剖析：`ModelRunner.run()` 内部逻辑**
1. **数据准备**：调用 `prepare_prefill` 或 `prepare_decode`。这里会将 Python 列表形态的 `input_ids`、`positions` 以及 `slot_mapping`（物理块映射关系）拍平成连续的 1D Tensor，并通过 `set_context()` 将元数据注入全局上下文，供后续的 FlashAttention 使用。
2. **网络前向传播 (`run_model`)**：
    - 如果是 Prefill 阶段，直接走正常的模型前向计算。
    - **注意点（CUDA Graph 捕获）**：如果是 Decode 阶段且未强制 Eager 模式，为了消除 Python 层面的 overhead 和 CUDA kernel 启动延迟，代码会利用提前捕获好的 CUDA Graph (`graph.replay()`) 执行网络。
3. **采样 (`self.sampler`)**：取出 Logits，根据 Temperature 并引入指数噪声做 Argmax 采样，得到生成的 Token IDs。

### 第三阶段：后处理 `self.scheduler.postprocess()`
拿到刚算出的 `token_ids` 后，需要更新系统的状态。进入 `Scheduler.postprocess()`：
- **对于 Prefill 请求**：由于是 Chunked Prefill，本轮可能只算了一部分。代码会将 `chunk_size` 累加到 `seq.num_cached_tokens` 中。
    - 如果 `seq.num_cached_tokens == len(seq)`，说明这个请求的 Prompt 终于算完了，将刚才生成的第一个 Token 追加进去，状态设为 `RUNNING` 转移到运行队列中（准备下一轮 Decode）。
    - 如果没算完，它依然留在 `waiting` 队列里，等待下一轮继续 Chunk 计算。
- **对于 Decode 请求**：直接调用 `seq.append_token(token_id)` 追加 Token。
    - 检查结束条件：如果遇到了 `eos`（且允许截断）或者达到了用户的 `max_tokens`，则将其状态标记为 `FINISHED`，并立即调用 `block_manager.deallocate(seq)` 回收显存，将其移出运行队列。

## 可能的问题
### PD 分离 kvcache 传输过程和设计逻辑？
逻辑：
- 调度器拿出本轮 prefill 完成后 kv cache，取到对应的 block table id
- 将这些元数据写入到 shm、然后用 isend 异步传输给对应的 decoder 节点
- decoder 通过 irecv 拿到 buffer 中然后 index copy 写回到本地 kv cache池中

### PD 分离通信的具体过程
在 `LLMEngine.step` 中，一次完整的 PD 分离迭代过程如下：
1. 异步发射 Decode 任务 (主进程 -> Decode 组)：
    - **操作**：主进程将 `run` 指令写入共享内存，并触发事件信号。
    - **异步性**：此处 `wait_ack=False`。主进程不需要等待 Decode 组确认接收，直接继续执行后续逻辑，以最大化掩盖 Decode 的启动延迟。
2. 同步执行 Prefill 任务 (主进程)：
    - **操作**：主进程同步调用 Prefill Runner 的 `run` 方法进行前向计算。
3. 结果收集屏障 (Decode 组 -> 主进程)：
    - **操作**：主进程调用 `recv_decode_res` 接收 Decode 组产生的 Token。
    - **同步性**：这是**同步**发生的。主进程必须在此阻塞，直到拿到结果，因为后续的调度逻辑依赖这些 Token 来更新序列状态。
4. KV Cache 传输信令 (主进程 -> 全体进程)：
    - **操作**：主进程确定需要传输的块，通过共享内存下发 `transfer_kv_cache` 指令。
    - **同步性**：此处 `wait_ack=True`。主进程必须等待 Decode 组确认已读取该指令，防止在下一轮迭代开始时，新的 `run` 指令过快覆盖掉共享内存中的传输指令。
5. P2P 数据传输 (GPU 之间)：
    - **操作**：Prefill 与 Decode 进程根据指令，在各自的 `transfer_stream` 上启动 NCCL 异步传输。

### 遇到难点
1. IPC 指令覆盖
	1. 主进程发指令太快，导致shm的指令还没被读取就已经被下一轮的新指令覆盖了
	2. 添加 ack 握手来解决这个问题
2. 计算重叠：
	1. 让 kv cache传输和计算流分开进行而不阻塞
3. 重构推理引擎，打破非 prefill 即 decode 的逻辑，将 Decode 任务分发给远端 GPU 组，同时利用本地算力并行处理 Prefill 任务

### 知道 vllm 实现的逻辑和自己的区别吗？
1. chunk prefill
	1. vllm：根据 GPU 剩余显存和设定的 token 预算，动态切分长文本，并与 Decode 任务混合在一个 Batch 中执行
	2. nano：定长的简单实现
2. PD 分离
	1. vllm：通常面向大规模集群，依赖 Ray 或者复杂的 RPC 框架来进行跨节点、跨进程的 Worker 协调。KV Cache 的传输涉及复杂的网络路由和分布式拓扑结构。
	2. nano：单机实现，torch dist + shm配合实现 IPC，cuda stream + 异步通信实现 PD 分离、kv cache传输等功能
3. prefix caching
	1. vllm：维护 radix tree，允许它在 **Token 级别** 进行非常灵活的前缀匹配，并配合 LRU 策略进行缓存驱逐。
	2. nano：链式哈希，实现 block 级别的 prefix

### 说一下 nccl 通信常用的方法
通信原语参考 [[torch.distributed#集合通信]]

不同并行策略下原语利用

| 并行策略            | 主要通信原语                                             | 说明                             |
| --------------- | -------------------------------------------------- | ------------------------------ |
| **数据并行 (DDP)**  | **All-Reduce**                                     | 同步梯度。                          |
| **张量并行 (TP)**   | **All-Reduce**, **All-Gather**, **Reduce-Scatter** | 矩阵乘法拆分，需要在 Layer 内部频繁通信。       |
| **流水线并行 (PP)**  | **Send/Recv** (P2P)                                | 层与层之间在不同 GPU 上，通过 P2P 传递激活和梯度。 |
| **FSDP / ZeRO** | **All-Gather**, **Reduce-Scatter**                 | 动态收集参数，更新后分片梯度，减少显存占用。         |
| **MoE**         | **All-to-All**, **All-Gather**                     | 根据 Token 路由进行数据重分布。            |

### 说说你的MoE支持EP吗？做了其他Parallel优化吗？
没做 EP，EP是指：针对MoE架构每次只能激活一少部分参数

我这里只做了 TP 的操作，主要对里面的一些线性层做的分割：
- Attention 的话
  - 对 Q、K、V 做了列切分
  - 对输出投影做了行切分
- FFN
  - gate、up做的列切分
  - down做的行切分
- embedding、lmhead：在 vocab 层面做的切分

设计思路的话，主要还是在不需要通信的地方切分计算，需要的地方插入通信原语
- Step 1：输入张量先通过列切分的 QKV 线性层，此时各张卡独立计算，没有任何通信开销。每张 GPU 只负责算出分配给自己的那几个 Heads。
- Step 2：各卡在本地独立完成 Attention 的 $QK^T$ 矩阵乘以及与 $V$ 的聚合。
- Step 3：局部 Attention 的结果随后送入行切分的 o_proj。由于行切分相当于把矩阵乘法的内积维度切开了，因此在局部计算完后，必须通过 torch.distributed.all_reduce 算子把各卡的部分和（Partial Sum）同步并累加起来，才能得到正确的完整输出供下一层使用。

### 说说知道其他什么 Parallel 的方法吗？
- 数据并行（DP / DDP）：最基础的并行方式。每张 GPU 持有完整的模型参数，通过切分数据 Batch 来并行，在反向传播时同步梯度。
- 全分片数据并行（FSDP / ZeRO）：为了解决 DDP 带来的显存冗余问题，将模型参数、梯度和优化器状态打散到不同的卡上。前向计算时用到哪一层就 All-Gather 哪一层的参数，是用通信带宽换取极致显存的核心技术。
- 流水线并行（Pipeline Parallelism, PP）：按模型的层（Layers）进行垂直切分，将不同层放置在不同 GPU 甚至不同节点上。通常会配合 Micro-batch 机制来减少流水线气泡（Bubble）。
- 专家并行（Expert Parallelism, EP）：这是针对 MoE（混合专家）模型专门设计的策略。将不同的 Expert 物理分配到不同的 GPU 上，利用 All-to-All 通信原语进行 Token 的路由转发和收集。在构建支持像 QWEN3-MOE 这类模型的推理引擎时，将 EP 和 TP 结合使用，是解决极度稀疏模型显存和计算瓶颈的最佳实践。
- 序列并行（Sequence Parallelism, SP）：通常与 TP 配合使用，将长文本的 Sequence 维度切分到不同的卡上（如 Ring Attention 机制），这在优化极长上下文（Long Context）推理时非常关键。

#### yitian710 cpu 架构是怎么样的，有什么特别吗？
yitian710 是 ARM 架构，一共有 128 个物理核，然后每个核有自己独立的L1 cache和 L2 cache  
大小分别是64KB和1MB，所以他不同核之间不会互相竞争缓存资源，多线程能力稍微强一点  

然后他支持 SVE 这些可拓展向量指令和 NEON 定宽并行的指令集，也就是说前者可以优化 decode 部分，后者定宽是可以优化 attention 计算时候的乘加融合操作的

#### yitian710 的 L1/L2 cache 是怎么样的？设计了什么 block size 策略
主要的 block 分块还是集中在 gemm 的时候，llama.cpp自带了 mn 层面的 tiling，然后取的向量乘的方式去做的 gemm，但实际上，向量乘的 K 会导致cache不够用，我们也没那么详细去测，大概的意思是通过 K 分块然后手动测几组数据比如 8、16、32、128，最后拿的 32

#### 统一 Q4_0 是什么意思？为什么要采取统一格式
Q_4是llama.cpp的量化策略之一，主要是把矩阵给32或者64一组分块，我记得是32分块吧，然后32共享scale去做的4bit量化

采取统一格式是因为 llama.cpp 默认会把特定张量用Q6_K_M 这种去做优化，但实际上做统一格式是为了内存重排，就是说你 load 完模型后把原始的 2D weight 给转化为 4D 的 (M/block，N/block，block，block) 的形式，所以统一量化格式后面 layout 会好写一点，并且你固定 4bit 也能适配 SIMD 

#### Q4_0 量化具体过程是什么？为什么不采取其他量化策略？

Q4_0 我理解就是把 weight 以 32、64 大小给打包，像一些其他策略比如 Q4_K_M 的话  
K就是 K means 的意思，你采取 K means 去量化，虽然这种情况下你的精度更高，但实际上我们测得精度在竞赛范围内完全是可以接受的，而且你 Q4_0 的内存效率更高，吞吐的提升远大于精度下降所以最终还是采取 Q4_0 去做

#### 为什么要做 layout ？ 
1. 优化空间局部性：Layout 确保 **计算上相邻的权重在内存物理地址上也相邻**。当 CPU 预取数据时，加载到缓存中的数据块能被立即全部用于计算，最大化了每次内存访问的利用率，减少了访问 DRAM 的次数。
2. 配合矩阵分块：通过将权重重排为符合 **Tile（分块）** 大小的结构（如文档测试的 32\*32 chunk_size 效果最好），确保每个 Tile 能完整放入 L1/L2 缓存。这样在计算该 Tile 期间，无需再次访问主存，显著降低了延迟。

#### 为什么 decode 提升比 prefill 高这么多
1. 首先是baseline起点低所以做了优化使得看起来相对提升高
2. 量化 Q4_0 优化了内存，显著提升了 Decode 的速度，因为 decode 是 memory bound
3. 利用 arm 架构做了 neon、sve2 指令集的优化
4. tiling 优化了访存，decode 是逐 token 生成的，对延迟敏感。通过矩阵分块优化 L1/L2 缓存命中率，减少了访问主存的延迟。由于 Decode 原本受限于内存访问延迟，缓存优化对其吞吐量的相对提升比例也高于 Prefill。
