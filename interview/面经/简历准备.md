# nano-vllm-extend
### 项目整体概述
nano-vllm 运行流程：
- LLM 引擎初始化
	- 配置文件与多线程启动
		- 加载 config
		- 创建 shm
	- model runner初始化
		- 创建 PD 通信组
		- 加载模型权重
		- kv cache显存分配
		- warm up捕获cuda graph
- generate 启动引擎推理
	- seq prompt 通过 tokenizer 转化为 token_ids 包装为 Sequence 对象
	- 加入 scheduler 的 waiting 队列
	- 调度 prefill_seq 和 decode_seq
		- decode 优先，显存不足先抢占已经被加入 decode_seq 队列的
		- 再 prefill 调度，如果是新请求，给 seq 初始化 page attention 物理块
	- PD 分离执行推理 (engine.step)
		- 异步 decode 推理
		- 同步 prefill 推理
		- 收集 decode 结果，此时同步起到了 barrier 的效果，确保 decode 能够推完（拿到本次 decode 的结果）
		- 当你 prefill 推完最后一个 chunk prefill 块的时候才推送 kvcache
	- postprocess
		- 更新 sequence 的 num_cache_token
		- 检查是否 chunk 推完转换状态为 running

### 可能的问题
#### PD 分离 kvcache 传输过程和设计逻辑？
逻辑：
- 调度器拿出本轮 prefill 完成后 kv cache，取到对应的 block table id
- 将这些元数据写入到 shm、然后用 isend 异步传输给对应的 decoder 节点
- decoder 通过 irecv 拿到 buffer 中然后 index copy 写回到本地 kv cache池中

#### PD 分离通信的具体过程
在 `LLMEngine.step` 中，一次完整的 PD 分离迭代过程如下：
1. 异步发射 Decode 任务 (主进程 -> Decode 组)：
    - **操作**：主进程将 `run` 指令写入共享内存，并触发事件信号。
    - **异步性**：此处 `wait_ack=False`。主进程不需要等待 Decode 组确认接收，直接继续执行后续逻辑，以最大化掩盖 Decode 的启动延迟。
2. 同步执行 Prefill 任务 (主进程)：
    - **操作**：主进程同步调用 Prefill Runner 的 `run` 方法进行前向计算。
3. 结果收集屏障 (Decode 组 -> 主进程)：
    - **操作**：主进程调用 `recv_decode_res` 接收 Decode 组产生的 Token。
    - **同步性**：这是**同步**发生的。主进程必须在此阻塞，直到拿到结果，因为后续的调度逻辑依赖这些 Token 来更新序列状态。
4. KV Cache 传输信令 (主进程 -> 全体进程)：
    - **操作**：主进程确定需要传输的块，通过共享内存下发 `transfer_kv_cache` 指令。
    - **同步性**：此处 `wait_ack=True`。主进程必须等待 Decode 组确认已读取该指令，防止在下一轮迭代开始时，新的 `run` 指令过快覆盖掉共享内存中的传输指令。
5. P2P 数据传输 (GPU 之间)：
    - **操作**：Prefill 与 Decode 进程根据指令，在各自的 `transfer_stream` 上启动 NCCL 异步传输。

#### 遇到难点
1. IPC 指令覆盖
	1. 主进程发指令太快，导致shm的指令还没被读取就已经被下一轮的新指令覆盖了
	2. 添加 ack 握手来解决这个问题
2. 计算重叠：
	1. 让 kv cache传输和计算流分开进行而不阻塞
3. 重构推理引擎，打破非 prefill 即 decode 的逻辑，将 Decode 任务分发给远端 GPU 组，同时利用本地算力并行处理 Prefill 任务

### 可能的问题
#### lora 微调？
用 lora 是因为有时间、计算成本  
lora思路：先通过对话和推理数据集解决理解能力再通过多领域数据集解决选择题的问题  
lora1 - 7个额外数据集一起调整，精度有提升，但是loss动荡剧烈且没能很好收敛  
lora2 - 仅使用对话和推理Ultra Chat数据集对原始模型进行微调，得到的lora2模型精度提升大概1%  

在 lora2 基础上使用剩余数据集进行第二轮微调，这时的分层训练使得 loss 曲线平缓、收敛速度也更快  

此外我们还验证了第三轮，就是把第二轮的 5 个领域数据集再拆开，结果发现效果反而查了一些  

#### yitian710 cpu 架构是怎么样的，有什么特别吗？
yitian710 是 ARM 架构，一共有 128 个物理核，然后每个核有自己独立的L1 cache和 L2 cache  
大小分别是64KB和1MB，所以他不同核之间不会互相竞争缓存资源，多线程能力稍微强一点  

然后他支持 SVE 这些可拓展向量指令和 NEON 定宽并行的指令集，也就是说前者可以优化 decode 部分，后者定宽是可以优化 attention 计算时候的乘加融合操作的

#### yitian710 的 L1/L2 cache 是怎么样的？设计了什么 block size 策略
主要的 block 分块还是集中在 gemm 的时候，llama.cpp自带了 mn 层面的 tiling，然后取的向量乘的方式去做的 gemm，但实际上，向量乘的 K 会导致cache不够用，我们也没那么详细去测，大概的意思是通过 K 分块然后手动测几组数据比如 8、16、32、128，最后拿的 32

#### 统一 Q4_0 是什么意思？为什么要采取统一格式
Q_4是llama.cpp的量化策略之一，主要是把矩阵给32或者64一组分块，我记得是32分块吧，然后32共享scale去做的4bit量化

采取统一格式是因为 llama.cpp 默认会把特定张量用Q6_K_M 这种去做优化，但实际上做统一格式是为了内存重排，就是说你 load 完模型后把原始的 2D weight 给转化为 4D 的 (M/block，N/block，block，block) 的形式，所以统一量化格式后面 layout 会好写一点，并且你固定 4bit 也能适配 SIMD 

#### Q4_0 量化具体过程是什么？为什么不采取其他量化策略？

Q4_0 我理解就是把 weight 以 32、64 大小给打包，像一些其他策略比如 Q4_K_M 的话  
K就是 K means 的意思，你采取 K means 去量化，虽然这种情况下你的精度更高，但实际上我们测得精度在竞赛范围内完全是可以接受的，而且你 Q4_0 的内存效率更高，吞吐的提升远大于精度下降所以最终还是采取 Q4_0 去做
